knitr

knitr is a package for dynamic report generation that enables the integration of R code into LaTeX, LyX, HTML, Markdown, AsciiDoc, and reStructuredText documents. The purpose of knitr is to allow reproducible research in R through the means of literate programming. Additional extensions also exist for python support. knitr is officially supported by RStudio.

When writing a report, I recommend using RMarkdown to save and execute code ina format that lends itself to generating quality reports that can be shared with collaborators or in a GitRepository. Instructions for how to format can be found here

Git

git is an open source version control system, however, most users are likely familiar with Github a great online tool for storing code and sharing with collaborators and/or the public. For daily work, however, I prefer to use GitHub Desktop, a graphical user interface for daily updates.

Begin by creating a new repository in whatever working directory you wish to keep your files stored. Doing so enables you to record any changes made to the repository, changes to scripts, updated notes, removed files, etc.

To commit changes that have been made, you will need to generate a Summary of the changes made and corresponding Description.

By committing the changes you confirm that any changes made to the files in your repository are saved. These changes are then also saved and recorded in the repositories History.

Before proceeding, I recommend establishing a new Branches. By default, every repository begins in the the main branch, this is the default branch for your finalized version. While working on your scripts, however, you may wish to save your work incrementally without finalizing the progress. For this you can establish a new branch that is a mirror copy of the main branch.

Branches can correspond can exist for any purpose, but one simple application is to ensure that each collaborator has their own branch. In this way, all collaborators can work on a project together, share the change without interfering with anyone elses progress. When ready to share your progress with others, you can Push your branch to the GitHub repository. This will upload the repository and all changes to the online storage so that other collaborators can see what youve done. If they wish to download a branch they can instead Pull from the GitHub repository to their local computer.

When a new branch has been uploaded that differs from the main, a Pull Request will be generated. Changes between two branches can be synchronized by Merging. In this process Git will compare each document and determine whether they can be successfully merged without conflict. In the case of a file removal or addition, this change will proceed un-interrupted. However, when there is a conflict, such as within the contents of a file, collaborators will be notified and asked to resolve the conflict. Once resolved, the two branches will be merged and the resulting synchronized brach is available for all collaborators to Pull from and continue their work.

Repository Structure

How you choose to structure your repository will depend on you, however, I recommend the following nested structure to make it easy to find and link to documents. Here are some examples of how I structure my repository.

## Data 
# store raw data and avoid altering files
dir.create(c("./data/raw"), showWarnings = FALSE, recursive = TRUE)
# store intermediate data for user manipulation
dir.create(c("./data/intermediate"), showWarnings = FALSE, recursive = TRUE)
# store final data for use in other documents
dir.create(c("./data/final"), showWarnings = FALSE, recursive = TRUE)
# store protocols and information concerning, sampling, labwork, etc.  
dir.create(c("./documentation/protocol"), showWarnings = FALSE, recursive = TRUE)
# store any references, in particular save all references in a single BibTex format
dir.create(c("./documentation/references"), showWarnings = FALSE, recursive = TRUE)
# write the bibtex file and save package citations
citPkgs <- names(sessionInfo()$otherPkgs)
knitr::write_bib(citPkgs, file="./documentation/references/references.bib")
# save workflow as scripts and use a prefix to identify the order of operation, e.g. 01-initialize.Rmd
dir.create(c("./scripts/analysis"), showWarnings = FALSE, recursive = TRUE)
# save miscellaneous scripts that can be called on during the workflow, e.g. plot_table.R
dir.create(c("./scripts/functions"), showWarnings = FALSE, recursive = TRUE)
